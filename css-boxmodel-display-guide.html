Chapter 1

The Box Model

What is a Box? Zero

Every element you put on a web page — a paragraph, a heading, an image, a link — becomes a rectangular box. CSS controls how these boxes look and how they sit next to each other.

Think of every HTML element as a parcel in a warehouse. It has contents inside, packing material around the content (padding), a cardboard wall (border), and empty space between it and other parcels (margin).

Visual: The Four Layers
MARGIN — space outside
BORDER — the wall
PADDING — inner cushion
CONTENT

The Four Parts Zero

Content — The actual text, image, or child elements. Width and height properties control this area by default.

Padding — Transparent space between the content and the border. It inherits the background color.

Border — A line (solid, dashed, dotted…) wrapping the padding + content area.

Margin — Transparent space outside the border. Always transparent; shows the parent's background through it.

/* The core box model properties */
width: 200px;
height: 100px;

padding: 20px;          /* all sides */
padding: 10px 20px;     /* top+bottom | left+right */
padding: 5px 10px 15px 20px; /* top right bottom left */

border: 2px solid #333; /* width | style | color */

margin: auto;          /* center horizontally */
margin: 0;             /* remove margin */
Chapter 2

Box Sizing — The Most Common Gotcha

The Default Problem Zero

By default, when you set width: 300px, the browser makes the content area 300px. Then padding and border get added on top. So the actual rendered size becomes wider than you expected.

/* content-box (default — confusing!) */
.box {
  box-sizing: content-box; /* default */
  width: 300px;
  padding: 20px;
  border: 2px solid;
  /* Actual rendered width = 300 + 20+20 + 2+2 = 344px */
}

/* border-box (intuitive — use this!) */
.box {
  box-sizing: border-box;
  width: 300px;
  padding: 20px;
  border: 2px solid;
  /* Actual rendered width = 300px exactly */
}
Pro Tip
Every professional project starts with this reset. Apply it globally: 
*, *::before, *::after {
  box-sizing: border-box;
}

Margin Collapse Intermediate

When two vertical margins touch, they don't add together — they collapse into the larger of the two. This only happens with margins (not padding), and only vertically, and only in certain situations.

h1 { margin-bottom: 40px; }
p  { margin-top: 20px; }
/* Gap between h1 and p = 40px (not 60px!) */

/* Margin collapse does NOT happen across formatting context boundaries */
/* Adding overflow:hidden or display:flex to parent prevents collapse */
Watch Out
Margin collapse only happens between block-level boxes in the same block formatting context. It does NOT happen between flex children, grid children, or elements separated by padding or border.
Chapter 3

The display Property

What display Does Zero

The display property is the master switch for how an element generates boxes. It controls two things simultaneously:

1. How the element participates in its parent's layout (its outer display type)

2. How it lays out its own children (its inner display type)

Core Concept
Modern CSS spec describes every display value as a combination of an outer type (block / inline) and an inner type (flow / flex / grid / table). The shorthand keywords you know are just precomposed pairs.

block vs inline Zero

Block elements stack vertically; inline elements flow with text
block box — takes full width, starts new line
inline elements flow together like words in a sentence.
another block box below
/* Block: div, p, h1-h6, ul, li, section... */
div {
  display: block;
  /* width defaults to 100% of parent */
  /* height fits content */
  /* width/height/margin/padding all work */
}

/* Inline: span, a, strong, em, img... */
span {
  display: inline;
  /* width and height are IGNORED */
  /* top/bottom margins are IGNORED */
  /* vertical padding doesn't push other elements */
}

inline-block Intermediate

The hybrid: flows inline with text but respects width, height, and vertical margin like a block. In modern spec terms this is inline flow-root.

.badge {
  display: inline-block;
  width: 80px;
  height: 24px;
  padding: 2px 8px;
  /* Now width/height work, but it still sits in text flow */
}

none vs contents Intermediate

display: none;     /* Element AND children disappear from layout entirely */
display: contents; /* Element's OWN box vanishes, but children remain */
/* 'contents' is like removing only the wrapper, not the contents */
/* Accessibility note: display:none removes from accessibility tree too */

The Two-Keyword Model Intermediate

According to the CSS Display Module Level 3, every non-special display value has two parts. The old single keywords are shorthand for these pairs:

Mental Model
Outer type answers: "How does THIS element relate to its siblings?"
Inner type answers: "How do MY CHILDREN get arranged inside me?"

flow-root — The Hidden Gem Advanced

display: flow-root makes an element a block-level box AND forces it to be a block formatting context (BFC) root. This is the clean, explicit way to contain floats and prevent margin collapse, instead of the old hacks.

/* Old hacks to create a BFC: */
overflow: hidden; /* works but clips content */
float: left;      /* changes layout behaviour */

/* Modern, explicit approach: */
.container {
  display: flow-root;
  /* Now contains floats, prevents margin collapse */
  /* No side effects! */
}

Formatting Contexts

What is a Formatting Context? Intermediate

A formatting context is the "room" in which boxes are arranged. The rules for how children are positioned depend entirely on which room they're in. Different rooms have different rules.

Four main formatting contexts
BLOCK FORMATTING CONTEXT (BFC) Children stack vertically. Floats are contained. Margins can collapse.
INLINE FORMATTING CONTEXT (IFC) Words and inline boxes wrap like text — left to right, then down.

Independent vs. Shared Contexts Advanced

When a box establishes an independent formatting context, its interior is isolated. Floats from outside can't intrude. Margins don't collapse across the boundary. The box grows to contain its floated children.

This is why adding overflow: hidden to a parent "magically" makes it contain floats — it creates a new independent formatting context as a side effect.

/* These all create an independent formatting context: */
display: flow-root  /* cleanest way */
display: flex       /* flex container */
display: grid       /* grid container */
overflow: hidden   /* side effect: creates BFC */
position: absolute /* also creates one */
float: left/right  /* also creates one */

Chapter 7

Advanced Topics

Blockification & Inlinification Advanced

Certain CSS situations force a box to change its outer display type automatically — this is called blockification or inlinification:

/* Blockification — forced outer type becomes block: */
position: absolute /* or fixed */
float: left         /* or right */
/* being a flex/grid child */

/* So this: */
span { position: absolute; width: 100px; }
/* is effectively treated as: */
span { display: block; position: absolute; width: 100px; }
/* width now works even though span is normally inline! */

The order Property Intermediate

Inside flex and grid containers, the order property changes visual order without changing the DOM (source) order. Use only for spatial reordering, not logical content reordering, because screen readers and keyboard navigation follow DOM order.

.container { display: flex; }

.item-a { order: 2; } /* appears third */
.item-b { order: 1; } /* appears second */
.item-c { order: 0; } /* appears first (default is 0) */

The Root Element's Box Advanced

The <html> element's display is always blockified — regardless of what you set. Its box establishes the initial containing block, which is the baseline for all percentage widths, absolute positioning, and viewport-unit calculations.

run-in Layout Advanced

display: run-in is a special, rarely-implemented value where an element merges into the following block, becoming inline content at the start of that block. Think dictionary definitions where the term is inline with its explanation.

dt { display: run-in; }
/* dt "runs into" the following dd, like: */
/* "term: definition text here" — all on one line */
/* ⚠️ Browser support is limited — use with care */

Anonymous Boxes Advanced

When the DOM structure doesn't match what a layout model needs, browsers silently insert anonymous boxes. These have no corresponding element — the browser generates them as structural fix-ups.

/* You write: */
<div style="display:table">
  <div style="display:table-cell">Cell</div>
</div>

/* Browser generates this structure internally: */
/* table → [anonymous table-row-group] → [anonymous table-row] → table-cell */
Advanced Insight
Anonymous boxes inherit CSS properties through their position in the box tree, not the element tree. This distinction matters when debugging why a CSS property seems to "jump" to an unexpected element.
